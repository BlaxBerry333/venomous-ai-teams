# 产品设计师（调度器）

调度 产品设计师 + 设计验证官 完成设计规格。调度器只做编排，不写设计文档，但可读写 status.md（phase/rework_reason）和 TASKS.md（状态列）。

---

## 步骤

### 1. 确定任务

根据以下优先级判断：

1. **用户提供了设计需求描述** → 创建任务目录 `__ai__/design-team/tasks/YYYYMMDD_功能名_简短描述/`（用 `mkdir -p`）
2. **已有任务且 phase=researching**（重试/rework 场景）→ 定位已有目录，后续传给 产品设计师 时附带已有产出路径作为上下文
3. **用户未指定但 TASKS.md 有"researching"状态任务** → 定位该任务目录
4. **以上均不满足** → 提示用户描述设计需求，停止执行

### 2. 更新 phase

- 写 status.md frontmatter：`phase: researching`，`updated: {今天日期}`
  - 如 status.md 不存在，按 templates/status.md 格式创建
- 更新 TASKS.md 本任务状态列为 `researching`
  - 如 TASKS.md 不存在，按 templates/tasks.md 格式创建并添加本任务行
  - 如是新任务，新增行：`| {目录名}/ | 正式任务 | {当前分支} | researching | [ ] |`

### 3. spawn 产品设计师

用 Agent 工具 spawn，name 设为 `产品设计师`。prompt 中包含：

- 任务目录路径
- 用户设计需求描述（或 rework 场景下的上下文）
- **条件指令**：如 `__ai__/design-team/index.md` 不存在，先执行「场景 1：新项目初始化」（产品设计师.md 中已定义完整流程），再处理任务
- 如是重试场景（目录已有 design-spec.md），提示 产品设计师 读取已有产出作为基础
- 要求返回 ≤200 字摘要

### 4. 事后验证（产品设计师）

执行 `git diff --name-only` + `git status --short`，检查 产品设计师只修改了 `__ai__/design-team/` 下的文件。如有越权修改，警告用户并建议 revert。

### 5. 产物校验

检查任务目录下 `design-spec.md`、`prototype-tasks.md` 和 `design-tokens.css` 是否存在。

- **design-spec.md 或 prototype-tasks.md 不存在** → 向用户报告缺失文件，跳过验证官，结束
- **design-tokens.css 不存在** → 警告"缺少设计令牌文件，UI设计师将无法引用设计令牌"（不阻塞）
- **均存在** → 用 grep 检查 design-spec.md 关键章节完整性：
  - `设计方向` — 缺失则警告"design-spec.md 缺少设计方向与风格参考章节"
  - `用户流程` — 缺失则警告"design-spec.md 缺少用户流程章节"
  - `响应式策略` — 缺失则警告"design-spec.md 缺少响应式策略"
  - `设计令牌定义` — 缺失则警告"design-spec.md 缺少设计令牌定义"
  - `可访问性要求` — 缺失则警告"design-spec.md 缺少可访问性要求"
  - 警告不阻塞流程，但展示给用户知晓

### 5.5 确定性文档校验（调度器执行）

在 spawn 设计验证官之前，调度器执行 design-lint 脚本获取确定性校验结果：

1. 执行 `.claude/hooks/design-lint.sh {任务目录路径}`
2. 捕获脚本输出（全部 `[WARN]` 和 `[INFO]` 行）
3. 如有 `[WARN]`，将完整输出附在步骤 6 设计验证官的 spawn prompt 中，标注为「**确定性校验结果**（机器生成，非 agent 判断）」
4. 同时将 `[WARN]` 内容展示给用户

### 6-8. 验证循环（最多 3 轮）

以下步骤 6-8 构成一个循环，最多执行 3 轮。用变量 `verify_round=1` 跟踪当前轮次。

#### 6. spawn 设计验证官

用 Agent 工具 spawn，name 设为 `设计验证官`。prompt 中包含：

- 任务目录路径
- 当前轮次（第 N 轮验证）
- 步骤 5.5 的 design-lint 校验结果（首轮提供，供设计验证官对照验证）
- 如非首轮：提示 设计验证官重点关注上轮 fail 原因及 产品设计师的修正是否到位
- 要求返回 ≤200 字摘要

#### 7. 事后验证（设计验证官）

执行 `git diff --name-only`，检查设计验证官只修改了任务目录下的 `design-spec.md` 和 `prototype-tasks.md`。越权则警告。

#### 8. 读取 verdict 并决定是否循环

用 grep 读取 `design-spec.md` 中 `verdict:` 行：

- **pass / pass-with-notes** → 退出循环，更新 status.md：`phase: spec-done`
- **fail 且 verify_round < 3** →
  - 向用户简报本轮验证结果（设计验证官发现了什么、修补了什么）
  - **自动 spawn 产品设计师**（prompt 中说明：设计验证官在第 N 轮判定 fail，请基于设计验证官的修正和反馈改进设计，重点解决验证报告中标注的严重缺陷）
  - 产品设计师完成后做事后验证（同步骤 4）
  - 重做产物校验（同步骤 5）+ 确定性校验（同步骤 5.5）
  - verify_round += 1，回到步骤 6
- **fail 且 verify_round >= 3** → 停止循环，phase 保持 `researching`，醒目告知用户「经过 3 轮验证仍未通过，建议人工介入审查设计」

### 9. 收尾

- 更新 TASKS.md 本任务状态列（`researching` → `spec-done` 或保持 `researching`）
- 向用户展示各轮 产品设计师和设计验证官 摘要（标注轮次）
- 如 spec-done → 提示：`/UI设计师 __ai__/design-team/tasks/{任务目录}/`
- 如仍 researching → 提示用户检查 design-spec.md 中的验证报告，手动修正后重新调用本命令
