# 程序员（调度器）

调度程序员完成代码实现。调度器只做编排，不写业务代码，但可读写 status.md（phase/rework_reason）和 TASKS.md（状态列）。

---

## 步骤

### 1. 定位任务

- **用户提供了路径** → 直接使用
- **用户未提供** → 执行 `ls -t __ai__/dev-team/tasks/` 找最近修改的目录，向用户确认"是否处理 __ai__/dev-team/tasks/XXX？"，确认后继续

### 2. 校验 phase

读取 status.md frontmatter 中的 `phase` 值。

- **phase ∈ {verified, rework, developing}** → 继续
- **其他值** → 提示用户当前 phase 不允许开发，停止执行

### 3. 更新 phase

写 status.md：`phase: developing`，`updated: {今天日期}`。更新 TASKS.md 状态列为 `developing`。

### 4. spawn 程序员

用 Agent 工具 spawn，name 设为 `程序员`。prompt 中包含：

- 任务目录路径
- **如 phase 来自 rework**：提示 程序员先读 status.md 的 `rework_reason` 字段确定返工来源（代码审查员或测试员），再读对应的 `review.md` 或 `test-report.md` 获取具体问题
- 要求返回 ≤200 字摘要

### 5. 事后验证（程序员）

执行 `git diff --name-only` + `git status --short`，检查以下文件未被修改：

- `__ai__/dev-team/index.md`
- `__ai__/dev-team/conventions.md`
- 任务目录下的 `design.md`、`review.md`、`test-report.md`
- `.claude/**`

如有越权修改，警告用户并建议 revert。

### 5.5 dev-plan.md 关键章节检查

用 grep 检查 dev-plan.md 是否包含关键章节标题：

- `代码变更摘要` — 缺失则警告"dev-plan.md 缺少代码变更摘要，代码审查员将无法定位审查范围"
- `交付核查` — 缺失则警告"dev-plan.md 缺少交付核查章节，无法确认交付完整性"

警告不阻塞流程，但展示给用户知晓。

### 6. 产物校验

读取 `dev-tasks.md`，检查是否有未勾选项（`[ ]`，兼容 `- [ ]` 和 `1. [ ]` 等格式）。

- **有未勾选项** → 不更新 phase，在输出中提示仍有未完成任务
- **全部勾选** → 继续

### 7. 更新 phase

写 status.md：`phase: dev-done`，`updated: {今天日期}`。

### 8. 更新 TASKS.md

更新本任务状态列：`developing` → `dev-done`。

### 9. 收尾

- 向用户展示 程序员摘要
- 提示下一步：`/代码审查员 __ai__/dev-team/tasks/{任务目录}/`
  - 如 rework_reason 以 `tester` 开头，额外说明：修复代码需经审查后再回到测试员复测
- 提醒用户：代码审查员会执行构建/测试/冒烟验证，发现运行时问题会回退到程序员修复
