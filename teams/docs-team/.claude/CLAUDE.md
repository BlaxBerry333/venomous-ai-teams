# AI 知识库文档专家团队

通过 Command 注入主上下文实现角色能力。直接在主对话中执行，不启动子进程。

## 权限执行机制

- `.claude/hooks/path-guard.sh` 在 Edit/Write 前检查目标路径，命中禁区（`.claude/**`、`app/**`、`__ai__/dev-team/**`、`__ai__/design-team/**`）一律 deny
- `.claude/hooks/doc-lint.sh` 在编程专家命令流程中做事后文档校验（链接、锚点、代码块、sidebar 同步）

跨 team 修改的处理见下方「行为约束」段。

## 角色调用规则

- 通过 `/角色名` 前缀命令调用角色
- 调用后角色能力在当前对话中持续生效，后续消息无需重复调用命令

## 主对话行为（未调用角色命令时）

当用户**没有使用 `/角色名` 命令**时，按以下优先级处理：

1. **只读操作直接执行**：阅读文档、搜索、分析结构、解释内容
2. **小改动直接执行**：修 typo、调整格式、改标点、修正 Markdown 语法错误（不涉及内容改动）
3. **内容创建/大幅修改先引导**：建议使用对应专家角色获得完整的文档能力
4. **用户坚持则执行**：用户明确表示不走角色流程时，直接执行，仅首次提醒

## 长会话 token 提示

docs-team 在主对话执行，历史中的文件读取与编辑反馈会持续累积。`/编程专家` 在任务执行（场景二/三/四）的摘要末尾会固定输出一行 `/compact` 建议。`/编程专家` 角色定义 < 5000 tokens，Claude Code 在 compact 后会自动重新附加，角色粘性保留，无需重新调用命令。未落盘的对话内容（如场景四审查报告）会被 compact 总结压缩，需持久保留请先落盘到 `__ai__/docs-team/` 下。

## 行为约束

- **不得修改** `.claude/` 目录下的任何文件
- **不得直接执行** `git push` 或 `git commit`（由用户手动操作）
- **不得修改** dev-team 或其他团队的任何文件：`app/**`、`__ai__/dev-team/**`、`__ai__/design-team/**` 由 `path-guard.sh` 硬拦截；`src/**` 因不同项目语义差异大未做硬拦截，由编程专家在 prompt 层自觉遵守"只改文档"
- **用户授权不是跨 team 豁免**：用户即便显式要求修改 docs-team 范围外的文件，docs-team 角色也应告知用户由他/她手动跑 `bash setup.sh` 切换 team，而非强改。以上约束由 `path-guard.sh` 硬拦截。切换 team 是"跨团队修改"的唯一正确方式；**Claude 不要自己跑 setup.sh**（交互式脚本，无法应答）

## 可用命令

| 命令 | 角色 | 用途 |
|------|------|------|
| `/编程专家` | 编程开发知识库专家 | 创建/编辑/整理/审查编程开发相关的学习文档 |

后续可扩展其他领域专家（如外语专家等）。
