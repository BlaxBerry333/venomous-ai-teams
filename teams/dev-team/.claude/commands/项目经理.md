# 项目经理（调度器）

调度 项目经理 + 验证官 完成设计。调度器只做编排，不写业务代码/设计文档，但可读写 status.md（phase/rework_reason）和 TASKS.md（状态列）。

---

## 步骤

### 1. 确定任务

根据以下优先级判断：

1. **用户提供了需求描述** → 创建任务目录 `__ai__/dev-team/tasks/YYYYMMDD_功能名_简短描述/`（用 `mkdir -p`）
2. **已有任务且 phase=designing**（重试/rework 场景）→ 定位已有目录，后续传给项目经理 时附带已有产出路径作为上下文
3. **用户未指定但 TASKS.md 有"designing"状态任务** → 定位该任务目录
4. **以上均不满足** → 提示用户描述需求，停止执行

### 2. 更新 phase

- 写 status.md frontmatter：`phase: designing`，`updated: {今天日期}`
  - 如 status.md 不存在，按 templates/status.md 格式创建
- 更新 TASKS.md 本任务状态列为 `designing`
  - 如 TASKS.md 不存在，按 templates/tasks.md 格式创建并添加本任务行
  - 如是新任务，新增行：`| {目录名}/ | 正式任务 | {当前分支} | designing | [ ] |`

### 3. spawn 项目经理

用 Agent 工具 spawn，name 设为 `项目经理`。prompt 中包含：

- 任务目录路径
- 用户需求描述（或 rework 场景下的上下文）
- **条件指令**：如 `__ai__/dev-team/index.md` 不存在，先执行「场景 1：新项目初始化」（项目经理.md 中已定义完整流程），再处理任务
- 如是重试场景（目录已有 design.md），提示项目经理 读取已有产出作为基础
- 要求返回 ≤200 字摘要

### 4. 事后验证（项目经理）

执行 `git diff --name-only` + `git status --short`，检查 项目经理只修改了 `__ai__/dev-team/` 下的文件。如有越权修改，警告用户并建议 revert。

### 5. 产物校验

检查任务目录下 `design.md` 和 `dev-tasks.md` 是否存在。

- **任一不存在** → 向用户报告缺失文件，跳过验证官，结束
- **均存在** → 用 grep 检查 design.md 关键章节完整性：
  - `现有代码分析` — 缺失则警告"design.md 缺少现有代码分析"
  - `对现有功能的影响分析` — 如涉及现有功能修改（非纯新增项目），缺失则警告"design.md 缺少影响分析"
  - 警告不阻塞流程，但展示给用户知晓

### 5.5 影响分析辅助验证（调度器执行）

在 spawn 验证官之前，调度器在主上下文中执行，为验证官提供确定性的辅助数据：

1. 从 design.md 的「需要修改的部分」提取文件路径列表
2. 对每个文件执行 `grep -rl "{文件名}" --exclude-dir=__ai__ --exclude-dir=.claude --exclude-dir=node_modules --exclude-dir=.git .` 搜索直接引用方
3. 将搜索结果汇总后附在步骤 6 验证官的 spawn prompt 中，供验证官对照 design.md 影响分析章节交叉验证

> 已知局限：grep 只能发现直接 import/require 关系。间接依赖链、运行时依赖、配置关联等无法通过 grep 发现。

### 6-8. 验证循环（最多 3 轮）

以下步骤 6-8 构成一个循环，最多执行 3 轮。用变量 `verify_round=1` 跟踪当前轮次。

#### 6. spawn 验证官

用 Agent 工具 spawn，name 设为 `验证官`。prompt 中包含：

- 任务目录路径
- 当前轮次（第 N 轮验证）
- 步骤 5.5 的 grep 搜索结果（首轮提供，供验证官对照 design.md 影响分析交叉验证）
- 如非首轮：提示 验证官重点关注上轮 fail 原因及 项目经理的修正是否到位
- 要求返回 ≤200 字摘要

#### 7. 事后验证（验证官）

执行 `git diff --name-only`，检查验证官只修改了任务目录下的 `design.md` 和 `dev-tasks.md`。越权则警告。

#### 8. 读取 verdict 并决定是否循环

用 grep 读取 `design.md` 中 `verdict:` 行：

- **pass / pass-with-notes** → 退出循环，更新 status.md：`phase: verified`
- **fail 且 verify_round < 3** →
  - 向用户简报本轮验证结果（验证官发现了什么、修补了什么）
  - **自动 spawn 项目经理**（prompt 中说明：验证官在第 N 轮判定 fail，请基于 验证官的修正和反馈改进设计，重点解决验证报告中标注的严重缺陷）
  - 项目经理完成后做事后验证（同步骤 4）
  - 重做产物校验（同步骤 5，确认 design.md 关键章节未在修改中丢失）
  - verify_round += 1，回到步骤 6
- **fail 且 verify_round >= 3** → 停止循环，phase 保持 `designing`，醒目告知用户「经过 3 轮验证仍未通过，建议人工介入审查设计」

### 9. 收尾

- 更新 TASKS.md 本任务状态列（`designing` → `verified` 或保持 `designing`）
- 向用户展示各轮 项目经理和验证官 摘要（标注轮次）
- 如 verified → 提示：`/程序员 __ai__/dev-team/tasks/{任务目录}/`
- 如仍 designing → 提示用户检查 design.md 中的验证报告，手动修正后重新调用本命令
