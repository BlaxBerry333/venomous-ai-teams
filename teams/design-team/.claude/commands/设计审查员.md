# 设计审查员（调度器）

调度 设计审查员 完成原型设计审查。调度器只做编排，不写审查报告，但可读写 status.md（phase/rework_reason）和 TASKS.md（状态列）。

---

## 步骤

### 1. 定位任务

- **用户提供了路径** → 直接使用
- **用户未提供** → 执行 `ls -t __ai__/design-team/tasks/` 找最近修改的目录，向用户确认"是否处理 __ai__/design-team/tasks/XXX？"，确认后继续

### 2. 校验 phase

读取 status.md frontmatter 中的 `phase` 值。

- **phase ∈ {prototype-done, reviewing}** → 继续
- **其他值** → 提示用户当前 phase 不允许审查，停止执行

### 3. 更新 phase

写 status.md：`phase: reviewing`，`updated: {今天日期}`。更新 TASKS.md 状态列为 `reviewing`。

### 4. spawn 设计审查员

用 Agent 工具 spawn，name 设为 `设计审查员`。prompt 中包含：

- 任务目录路径
- 要求返回 ≤200 字摘要

### 5. 事后验证（设计审查员）

执行 `git diff --name-only` + `git status --short`，检查 设计审查员只修改了任务目录下的 `design-review.md`。如有越权修改，警告用户并建议 revert。

### 6. 读取审查结论

用 grep 读取 `design-review.md` 中 `review_result:` 行：

- **pass** →
  - 写 status.md：`phase: done`，`updated: {今天日期}`，`rework_reason: ""`
  - 更新 TASKS.md 状态列为 `done`
- **rework-minor** →
  - 写 status.md：`phase: rework`，`updated: {今天日期}`，`rework_reason: reviewer-minor`
  - 更新 TASKS.md 状态列为 `rework`
- **rework-critical** →
  - 写 status.md：`phase: rework`，`updated: {今天日期}`，`rework_reason: reviewer-critical`
  - 更新 TASKS.md 状态列为 `rework`

### 7. 收尾

- 向用户展示 设计审查员摘要
- **如 done** → 告知用户设计任务完成。提示：
  - 可在浏览器中打开 `mockup.html` 查看最终设计
  - dev-team 开发时可参考 `__ai__/design-team/tasks/{任务目录}/mockup.html` 和 `design-tokens.css`
- **如 rework** → 告知用户需修复审查问题，提示：`/UI设计师 __ai__/design-team/tasks/{任务目录}/`
  - 列出审查中发现的 🔴 严重问题数量和 🟡 中等问题数量
