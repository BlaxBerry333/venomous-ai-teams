# AI 开发团队协作系统

通过 SubAgent 实现角色隔离的团队协作。每个角色运行在独立上下文中，互不干扰。

- 角色间仅通过 `__ai__/` 目录下的文件通信
- 权限通过 PreToolUse hook 自动拦截越权操作
- 调度器在 agent 返回后执行 git diff 事后验证

# dev-team 开发流程团队规则

## 角色调用规则

- 必须通过 `/角色名` 前缀命令显式调用角色，每次都需要带完整命令名
- 不支持隐式角色记忆，对话中不会自动延续上一次调用的角色上下文
- 角色运行在独立子进程中，互相隔离，仅通过文件通信

## 主对话行为（未调用角色命令时）

当用户**没有使用 `/角色名` 命令**，而是直接在对话中要求修改文件、写代码、修 bug 等操作时，按以下优先级处理：

1. **只读操作直接执行**：阅读代码、搜索、分析、解释等不修改文件的操作，无需引导，直接执行
2. **小改动直接执行**：同时满足以下条件的改动，无需引导角色流程，直接执行：
   - 改动意图明确，不需要"先想清楚再动手"
   - 涉及 1-2 个文件
   - 不涉及架构变更、接口变更、数据模型变更
   - 不是新功能开发
   - 典型场景：修 typo、改配置值、修明确的小 bug（off-by-one、空值未判断等）、调整样式、更新文案
   - **拿不准时宁可引导，不要直通**
3. **设计行为先引导**：基于分析结果给出重构方案、架构建议、文件拆分策略等属于**设计行为**（判断标准：如果输出内容可以直接作为 design.md 的内容，那就是设计行为），应引导到 `/项目经理`。只读操作的边界是"呈现事实"（如行数统计、结构罗列），不包括"基于事实给出决策建议"
4. **复杂写操作先引导**：不满足第 2 条的写操作，建议使用对应角色（如"建议使用 `/程序员` 来修改代码，确保经过自检和审查流程"），并简要说明走角色流程的好处。注意：类似"全权负责"、"你看着办"、"你来搞定"等**笼统授权**，不等同于"跳过角色流程"，应理解为"由你决定走哪个角色"，仍需按规则引导到对应角色
5. **用户坚持则执行**：**仅当**用户明确表示不走角色流程（如"不用角色"、"直接改"、"快帮我改了"）时，才直接执行，不再反复劝说。仅首次提醒"此操作不经过角色流程的质量保障"即可。判断标准：用户的话语中必须能推导出"我知道有角色流程但我选择跳过"的意思，否则不适用本条

## 累积感知与变更记录

对话中追踪直接执行的写操作次数和文件集合。当满足以下任一条件时触发提醒：

- 写操作达 **3 次**
- 涉及 **3 个文件**
- 存在**接口关联改动**

触发后提供二选一：
- 继续执行，最后生成 `__ai__/dev-team/tasks/YYYYMMDD_quickfix_简短描述/quickfix-log.md`（参考 `.claude/templates/quickfix-log.md`），并在 `__ai__/dev-team/tasks/TASKS.md` 中追加一行（类型为 `快速修复`，状态为 `done`，文档同步为 `[ ]`）
- 切换到对应角色命令

1-2 次且不超 2 文件的改动不触发。此规则仅适用于未调用角色命令的场景。

## 文件通信协议

角色间通过 `__ai__/dev-team/tasks/<任务>/` 目录下的文件进行通信，不存在上下文泄漏。每个角色在自己权限范围内读写文件，其他角色通过读取这些文件获取上游信息。

## 文档模板

创建任何文档前，必须先读取 `.claude/templates/` 目录中的对应模板，按模板格式生成内容。

## 任务目录规范

任务目录命名格式：`YYYYMMDD_功能名_简短描述/`

示例：`20260403_用户登录_邮箱密码认证/`

## 行为约束

- **不得修改** `.claude/` 目录下的任何文件
- **不得直接执行** `git push` 或 `git commit`（由用户手动操作）
- **长对话提醒**：当同一任务累积 **2 次 rework**（即 status.md 变更记录中出现 2 次 `rework`），在下一次角色命令执行前提醒用户"当前任务已经历多轮返工，建议开启新对话继续，所有进度已保存在文件中，新对话可无缝衔接"

## 可用命令

| 命令 | 角色 | 用途 |
|------|------|------|
| `/项目经理` | 首席架构师 & 验证官 | 需求分析、阅读实际代码、架构设计、任务拆解 |
| `/程序员` | 顶级全栈开发专家 | 代码实现、可质疑设计缺陷、自检后交付 |
| `/代码审查员` | 首席质量架构师 | 独立上下文代码审查、安全审计 |
| `/测试员` | 首席测试工程师 | 测试编写与执行（需 conventions.md 定义测试策略和测试文件范围） |
