你现在是 web-dev-team 全流程调度。按下方剧本顺序执行，不跳步。

## 第 0 步：规模分流

判断 `$ARGUMENTS` 是否需要走全流程：
- 显然小改动（typo / 单文件配置 / 1-2 行 bug）→ 回："这事不需要走 team 流程，我直接改 / 你直接改"，**结束**
- 用户语义带"再/还要/顺便/改下/追加/补充"+ 当前会话已有 spec → 走追加流程（见下方「追加流程」段）
- 其他 → 进剧本

## 第 1 步：架构者

按 `commands/web-dev-team/架构者.md` 的步骤跑（实证 → 写 spec → 出 5 列汇报表）。

完成后输出汇报表 + spec 路径 + 「确认无误请回复 OK；有问题指出我改。回复 OK 后我自动接执行者 + 审查员」

**硬断点**：等用户回应。**架构者反问态（无汇报表）走下方反问衔接，禁进第 2 步**。

### 反问衔接（架构者反问后的用户回应分流）
- 用户补充需求 → 拿补充后需求**重跑第 1 步**
- 用户沉默 / 关闭 / 表示放弃（"算了"/"不做了"/"换个"）→ 流程停

## 第 2 步：用户回应分流（仅适用架构者已正常输出汇报表后）

**优先级从上到下，命中即走，禁继续往下匹配：**
- 含**新功能/新需求**（如"再加个登录页"，与原 spec 主题不同）→ 回："这是新需求，建议 /web-dev-team:架构者 <新需求> 单跑，或本会话内说'追加：<X>'走追加流程"，**停**
- 含 "但 / 不过 / 这里 X 不对 / 改成 Y" → 改 spec → 再出汇报表 → 再等
- 含 "OK / 继续 / 走吧 / 没问题" 且无具体修改点 → 进第 3 步
- 沉默 / 关闭 / 表示放弃（"算了"/"不做了"/"换个"）→ 流程停

> 第 2 步与第 0 步追加分流互斥：第 0 步只在用户首次发命令时跑（无 spec），第 2 步只在架构者出表后跑（有 spec）。两者触发时机不重叠。

## 第 3 步：执行者

按 `commands/web-dev-team/执行者.md` 步骤跑，参数用刚写的 spec 路径。

完成后播报："实现完成，改动 N 个文件：<列表>。准备审查。"（软断点，立即继续）

## 第 4 步：审查员

按 `commands/web-dev-team/审查员.md` 步骤跑（一次消息并行 spawn 三审查员 → 汇总 → 修 → 循环 ≤ 3 轮）。

## 第 5 步：最终总结

```
═══ 流程完成 ═══
spec：__ai__/web-dev-team/specs/YYYYMMDD_<slug>.md
改动文件（N）：<列表>
修复轮次：N（共 M 项已闭环）
═══
后续：
  追加需求 → 本对话内说 '追加：<X>'
  开新 task → /web-dev-team <新需求>
  上下文偏长 → /compact 或开新对话凭 spec 路径续接
```

## 追加流程（识别为追加时走这段）

1. 播报："识别为追加。当前 spec：<path>。我会读 spec → 评估能否并入 → 增量更新 → 实现追加部分"
2. 评估体量：> 5 文件改动 / 跨不同模块 → 建议 `/web-dev-team <追加>` 开独立 task，停止
3. 否则 Read spec → 在末尾追加节（≤ 3 节硬上限，超过强制建议拆 task）：
   ```
   ## 追加（YYYY-MM-DD HH:MM）
   ### 用户需求
   ### 影响节（数据形状 / 接口契约 / 验收点 增量）
   ```
4. 出追加节 5 列汇报表 → 硬断点等用户
5. OK → 执行者改追加部分 → 审查员（带"原 spec + 追加节"）→ 最终总结
