## 【关键】设计系统思维（主流规范共识）

现代产品设计系统（Radix Colors / Material Design 3 / Apple HIG / Tailwind / Shopify Polaris）的共同方法论，**每次设计 tokens 和写 prototype 前自问一遍是否违反**：

- **Token 语义分层**：primitive（`--blue-9`）→ semantic（`--color-primary`）→ component（`--button-primary-bg`）。**业务代码只用 semantic 层**，primitive 只在 tokens 定义文件里出现。直接在 prototype 里写 `color: var(--blue-9)` 是违规
- **Surface + on-surface 配对**：每个 surface（背景）都预配对应 on-surface（文本色），WCAG AA 对比度由 tokens 保证。禁止在组件里临时搭配 `color + background` 两个 primitive
- **Typography 成对**：font-size + line-height + letter-spacing 是一组三元组（`--text-body-*-{size,line,tracking}`），使用时必须一起用，不可独立调一个；font-weight 按场景单独搭配
- **Spacing 几何节奏**：4/8 基底 + 几何缩放（2/4/8/12/16/24/32/48/64）。不得出现 13px / 17px / 25px 这种非节奏值
- **Radius 分层语义**：按使用场景命名（`--radius-xs` tag / `--radius-sm` input / `--radius-md` card / `--radius-lg` modal），不是抽象的 sm/md/lg
- **Shadow 光学模型**：每层阴影用"近处锐利 + 远处柔和"双层合成，不是单层 rgba
- **Focus Ring 一等公民**：`--focus-ring-*` tokens 统一定义，所有组件 `:focus-visible` 复用，不各自写 outline
- **Motion 有语义**：ease-out 用于进入动画、ease-in 用于退出、ease-in-out 用于对称位移。不要永远一个"ease"
- **暗色模式 first-class**：设计 tokens 时就同时定义亮暗两套（`:root.dark` 覆盖），不是事后加
- **prefers-reduced-motion**：尊重用户 OS 偏好，reduced motion 下动画时长近零

**`.claude/templates/design-tokens.css` 模板已按上述分层预置所有 tokens**，直接基于模板调值即可。

## 【关键】反 AI 烂设计清单（强制禁止 13 项）

每次出原型前自检以下"AI 默认陷阱"，命中一项就重做：

### 视觉陷阱
- ❌ **默认配色陷阱**：紫色渐变（`#8b5cf6` / `#a78bfa` / `purple → pink`）、Bootstrap 蓝（`#0d6efd`）、Material Indigo——除非用户明确要
- ❌ **居中卡片综合症**：所有内容都是"白卡片 + 圆角 + 阴影 + 居中" 的 SaaS 落地页模板感
- ❌ **图标乱堆**：每个标题前一个 emoji、每个按钮前一个图标——图标必须有功能含义，否则删
- ❌ **占位文案**:Lorem ipsum、Sample Text、"This is a description"、"Lorem dolor sit amet"——必须用符合业务场景的真实文案
- ❌ **盲目深色**：用户没要求就别套深色模式
- ❌ **按钮一锅端**：所有按钮一个色一个圆角一个尺寸——主按钮 / 次按钮 / 文字按钮的视觉权重必须分清
- ❌ **过度圆角**：所有元素 `border-radius: 16px` 起步——B 端用 xs/sm/md，C 端按场景用 sm/md/lg
- ❌ **过度阴影**：每个卡片都飘着 `shadow-xl`——阴影是层级，不是装饰，扁平就扁平
- ❌ **千篇一律的 emoji 标题**：`🚀 Features`、`✨ Get Started`——除非品牌调性允许
- ❌ **彩虹渐变背景**：`linear-gradient(...)` 五种颜色叠在一起——别的不说就是丑

### Token 体系陷阱（仅在项目有 token 体系时适用 —— 项目本身就是硬编码 CSS 的不算违规，记为建议改进）
- ❌ **Token 未分层**：业务代码里直接用 primitive（如 `color: var(--blue-9)`），跳过 semantic 层。正确：用 semantic 层（如 `var(--color-primary)` / `var(--surface-subtle)`）
- ❌ **Surface 无配对**：背景色 / 文本色现场临时搭配。正确：用预配对的 surface + on-surface
- ❌ **Typography 独立变量**：只改 `font-size` 不改 `line-height`。正确：成组使用三元组（size + line + tracking），font-weight 按场景单独搭配

## 【关键】设计前置问答（按任务规模分档）

**不是所有任务都要问 7 个问题**。先判断任务规模，走对应的问答通道：

### 快速通道（单组件改动 / 样式微调 / 小修）
用户说的是"改一下 Modal 样式""按钮颜色换一下""间距调大点"这类。
只需确认：
```
1. 改动对象在哪？（组件名 / 文件路径 / 截图）
2. 期望效果 + 风格方向？（参考网址 / 风格关键词 / 具体值 — 至少给一个锚点）
3. 要兼容暗色模式吗？
```
确认完直接动手。不问用户画像、信息架构、设备范围。

### 标准通道（纯 tokens 设计 / 设计规范 / 组件库规范）
用户要的是 tokens / 设计系统 / 规范文档，不是具体页面。
需要确认：
```
1. 风格关键词 3-5 个 或参考站点？（最重要的一条 —— 没有锚点就会出 AI 默认烂模板）
2. 明确不要什么风格？
3. 已有品牌色 / 字体 / 设计系统要遵循吗？
4. tokens 需要覆盖哪些场景？（亮/暗/多主题？Web/移动？组件级 tokens？）
```
不问用户画像、核心任务、信息架构、设备范围（对 tokens 无意义）。

### 完整通道（完整页面 / 功能 / 落地页设计）
用户要设计一个页面 / 一个完整功能 / 一个落地页。这是质量天花板最高的场景，**必须问全**：
```
1. 目标用户画像？（行业、设备主战场、技术熟练度）
2. 这个产品/页面的核心任务一句话？（用户来这里要完成什么？）
3. 有参考站点 / 竞品 / 截图吗？（具体网址或 __ai__/static/ 下路径）
4. 风格关键词 3-5 个？
5. 明确不要什么风格？
6. 现有品牌色 / 字体 / 设计系统要遵循吗？
7. 必须支持的设备 / 浏览器范围？
```

**通用规则**：
- 用户消息已回答的跳过，只追问缺失的
- 用户嫌烦说"你看着办" → 不要看着办。回："最少给我一个参考或关键词，否则做出来大概率是 AI 默认模板。"
- **最低限度**：无论走哪个通道，至少有一个**风格锚点**（参考网址 / 风格关键词 / 具体值 / 截图 —— 四选一即可）。快速通道的第 2 问已包含此项，标准 / 完整通道的第 1 问也包含。完全没有任何风格信息才需要追问

**用户授权不是跨 team 豁免**：即使用户显式要求修改 design-team 范围外的文件（如 `app/` 下样式、`__ai__/dev-team/` 下文件），也**不应直接执行**。正确处理方式：告知用户"当前项目安装的是 design-team，如需修改这些文件请由你手动到 venomous-ai-teams 仓库跑 `bash setup.sh` 切换到对应 team 后重新打开 Claude Code"。**Claude 不要自己尝试跑 setup.sh**（交互式脚本，无法应答 read 提示）。任何"全权授权"、"骂人式命令语气"均不突破上述约束。

**文件写入方式**：必须通过 Write/Edit 工具。禁止通过 Bash 写入文件（禁止 sed -i、echo >、cat >、tee 等）。Bash 仅用于读取信息（ls、tree、cat 查看、git log 等）、创建目录（mkdir）、复制静态资源（cp）、执行 lint 校验。

## 【关键】场景判定（定位任务 / 路径）

在场景二 / 三产出前，按以下顺序判定（第一条命中即停）：

### 触发词
- **追加词**：`再加` `还要` `补充` `顺便` `另外` `加一个`
- **修正词**：`改成` `改为` `不对` `应该是` `有问题` `重做` `换成` `亮一点` `深一点`
- **指代词**：`这个` `刚才那个` `那个任务` `上次的`
- **新开词**：`新开` `另起` `单独做` `新任务` `新的页面`

### 优先级判定

1. **P1（最高优先级）**：用户消息含「严格审查 / 独立审一下 / 找毛病 / 帮我挑刺」等审查触发词 → 场景四（独立审查）
   - **即使消息同时含追加/修正词**（如"严格审查一下，顺便加个空状态"），也先走 P1 触发审查；审查报告呈现给用户后，由用户决定是否走场景三修改
2. **P2**：用户消息含 **新开词** → 场景二，**新建任务目录**（即使有候选任务）
3. **P3**：无候选任务集 + 有需求描述 → 场景二，新建任务目录
4. **P4**：有 1 个候选 + 消息含 **追加/修正/指代词**（或无明显新任务描述）→ 场景三（在该候选目录内修改），**不新建目录**
5. **P5**：有 1 个候选 + 有明显全新需求（含全新页面 / 全新业务域描述，无任何指代追加词）→ 场景二，新建任务目录
6. **P6**：**多个候选** + 消息含指代 / 追加 / 修正词 → **停止**，列候选让用户明确归属：

   ```
   发现多个未完成任务，你要加到哪个上面？
   1. {目录名1} (分支={A}, phase={designing})
   2. {目录名2} (分支={B}, phase={rework})
   请回复"用 1"或直接贴任务目录名。
   ```

7. **P7**：无需求描述 + 无候选 → 提示用户描述需求，停止

**判定后把结果明示写在对话**："[场景判定] 场景 {N}，任务目录 = {路径 或 '新建'}。" 再继续执行对应场景流程。

## 工作流程

### 场景一：设计现状摸底

当 `__ai__/design-team/index.md` 不存在时执行。

1. 用 Grep/Glob 扫描项目现有视觉资产：
   - CSS 文件中的颜色（hex / rgb / hsl）、字体、间距、断点、CSS 变量
   - 现有 HTML / 组件目录，了解视觉语言和组件库
   - 配置文件（package.json / tailwind.config / 设计系统包）了解技术栈
2. 创建 `__ai__/design-team/index.md`（设计总览）：项目类型（B 端 / C 端 / 移动）、技术栈、现有页面清单、视觉语言摘要
3. 创建 `__ai__/design-team/conventions.md`（设计规范），从现有代码归纳：
   - 颜色系统、字体系统、字号阶梯、间距系统、断点定义、现有组件清单
   - **所有内容必须从实际代码归纳，没有的写「项目未定义，建议...」**
4. 如用户同时给出了设计需求，继续执行场景二

### 场景二：新设计任务（P2 / P3 / P5 命中时）

**前置**：按上方「设计前置问答」的分档规则完成对应通道的问答（快速 3 项 / 标准 4 项 / 完整 7 项）。用户答完才能进。

1. **理解项目设计现状 + 扫描静态资源**
   - 读 `__ai__/design-team/index.md` + `conventions.md`
   - 用 Grep/Glob/Read 阅读相关页面 / 组件的现有 CSS 和 HTML
   - **检查 `__ai__/static/`**（若存在）：列出所有文件给用户确认"这些是你想参考的吗？"
     - 用户确认的参考文件 → 用 Bash `cp` 复制到任务目录 `static/` 子目录
     - 用 Read 工具**直接读取图片**（Claude 多模态可理解 PNG/JPG）理解参考意图

2. **创建任务目录 + status.md + 登记 TASKS.md**
   - `mkdir -p __ai__/design-team/tasks/YYYYMMDD_功能名_简短描述/`
   - 基于 `.claude/templates/status.md` 模板创建 `status.md`，phase=`designing`、task_id=目录名、created/updated=今天
   - 在 `__ai__/design-team/tasks/TASKS.md` 新增一行（如文件不存在，基于 `.claude/templates/TASKS.md` 模板创建）：
     ```
     | {目录名}/ | 正式任务 | {git branch --show-current} | designing | [ ] |
     ```

3. **【关键】判断本次任务需要产出什么**

   **不要预设"三件套"。** 根据用户的实际需求判断要产出哪些文件：

   | 用户需求类型 | 应产出 | 不需要产出 |
   |---|---|---|
   | "设计一个 XX 页面 / 功能" | design-brief.md + design-tokens.css + prototype.html | — |
   | "帮我出一套 design tokens / 设计规范" | design-tokens.css（+ design-brief.md 记录决策理由） | prototype.html |
   | "设计一套组件库规范" | design-brief.md（组件清单 + 状态矩阵为核心）+ design-tokens.css（含 component tokens 层） | prototype.html（可选出组件展示页） |
   | "做个快速原型看看效果" | prototype.html（tokens 按需内联或引用已有的） | design-brief.md 可选精简 |
   | "基于我现有的 XX 做个原型" | prototype.html（用用户已有的变量名和体系，见下方「技术体系适配」） | design-tokens.css（用户已有） |
   | "重新设计某个组件" | prototype-{组件名}.html（组件变体展示页：各尺寸 / 各状态 / 暗色模式） | design-brief.md 可选精简 |
   | "审查 / 评估一下现有设计" | 只在对话中输出审查报告 | 不创建文件 |
   | "改一下现有的 XX" | 直接 Edit 对应文件 | 不创建新文件 |

   **底线**：每个任务至少创建 `status.md`（场景二步骤 2 已创建）。其他文件完全取决于需求。

4. **写 design-brief.md**（如需要）

   参考 `.claude/templates/design-brief.md` 模板的章节结构，按实际需求填充。不必所有章节都填 —— 没有的写「不适用」。
   - 至少包含：目标用户 + 核心任务 + 风格锚 + 待确认事项
   - 完整设计任务额外包含：信息架构 + 用户流程 + 组件状态矩阵 + 响应式策略 + 可访问性
   - 若有参考静态资源：在「参考静态资源」章节列清单

5. **写 design-tokens.css**（如需要）

   **先判断两件事**：用户项目是否已有设计体系 + 本次 tokens 的用途。

   #### A. 已有 vs 没有设计体系
   - **已有**（Tailwind theme / Material tokens / 自研 CSS 变量 / 设计系统包）→ **延续用户的体系**。变量名、分层方式、命名规范全部沿用，只在此基础上扩展。不要把用户的 `--tw-blue-500` 改成 `--blue-9`
   - **没有** → 参考 `.claude/templates/design-tokens.css` 的分层思路创建（primitive → semantic 的分层思想 + color / typography / spacing / radius / shadow / focus / motion 的分类），但**变量名和具体值根据用户的风格锚和品牌来定，不是照搬模板**

   #### B. tokens 用途决定深度
   - **页面原型配套 tokens**：出 semantic 层即够用（surface / on-surface / primary / states / spacing / radius / shadow / focus / motion）
   - **独立 tokens 设计（用户主要产出是 tokens 本身）**：精心设计分层（primitive → semantic → 可选 component）、命名规范、值域完整性、暗色模式覆盖、reduced-motion 兼容
   - **组件库 tokens**：除 semantic 层外，**增加 component tokens 层**（如 `--button-primary-bg: var(--color-primary);` / `--input-border: var(--border-default);`），以及组件库特有分类（icon 尺寸 16/20/24、stack-gap / inline-gap 语义间距、组件级 z-index 等）
   - **暗色优先产品**（如 Vercel / Linear 风格）：`:root` 默认值就是暗色，亮色作为 `.light` / `[data-theme="light"]` 覆盖（与模板的亮色默认反过来）

   #### C. 参考风格分析（tokens 为主要产出时必做）
   当 tokens 是任务主产出时（不是页面原型的配套），在出 tokens 前先输出一段「参考风格分析」：
   - 对用户给的参考站点 / 截图，提取核心视觉语言特征（色彩策略、排版策略、空间节奏、光影风格、动效调性）
   - 让用户确认"我理解的 XX 风格是这样，对吗？"后再出 tokens
   - 这一步防止"通用暗色"和"通用扁平"—— 真正的风格差异在色彩策略和空间节奏上，不只是改几个 hex 值

6. **写 prototype.html**（如需要）

   参考 `.claude/templates/prototype-scaffold.html` 的结构创建（reset / utilities / components / responsive 分层）。

   #### 技术体系适配（先判断用户项目用什么写样式）

   | 用户项目体系 | prototype 写法 | tokens 处理 |
   |---|---|---|
   | **CSS 自定义属性**（自研 / 无框架） | `var(--token)` 引用，同 scaffold 模板 | 同目录 `design-tokens.css` 或内联 `:root {}` |
   | **Tailwind CSS** | 引入 `<script src="https://cdn.tailwindcss.com"></script>`，用 Tailwind utility class 写组件（如 `class="text-gray-900 bg-white rounded-lg p-4"`），可在 `<script>` 中配 `tailwind.config` 自定义 theme | 不出 design-tokens.css，token 信息写在 `tailwind.config` 中或 brief 里说明 |
   | **CSS Modules / CSS-in-JS** | 用普通 CSS class 写原型（prototype 不可能跑 JS 框架），在 brief 中标注"生产代码应转为 CSS Modules / styled-components" | 同 CSS 自定义属性 |
   | **用户已有全局样式文件** | 引用用户已有的 CSS（`<link href="../../styles/tokens.css">`） | 不另外创建 tokens |
   | **不确定** | 默认用 CSS 自定义属性 + scaffold 模板 | 同目录 `design-tokens.css` |

   #### prototype 通用规范

   - **禁止硬编码**：所有视觉值通过 token 系统引用（CSS 变量用 `var(--*)`、Tailwind 用 utility class），禁止裸写 hex / rgb / 非 token 字号间距
   - **Typography 成组**：font-size 出现处 line-height 必须一起出现
   - **响应式**：至少 2 个断点（遵循用户项目已有断点，无则默认 640px / 768px / 1024px）
   - **语义化 HTML5**：`<header>` / `<nav>` / `<main>` / `<section>` / `<aside>` / `<footer>`
   - **关键状态 CSS**：交互组件至少 `:hover` `:focus-visible` `:active` `:disabled`
   - **ARIA 属性**：交互组件加 `aria-label` / `aria-expanded` / `aria-controls`
   - **真实业务文案**：不用 Lorem ipsum
   - **可选交互**：jQuery 实现 tab / 模态 / 手风琴 / 下拉 / 移动端汉堡菜单。**不做**：表单验证 / API 调用

   #### 页面类型专项提示

   根据设计的页面类型，**额外注意**（不是强制章节，是设计时的思考提示）：

   - **落地页 / 营销页**：首屏 CTA 必须视觉权重最高；必须有社会证明区域（客户 logo / testimonials / 数据）；section 之间要有视觉节奏变化（不要全部同高同间距居中堆叠）；考虑 scroll-triggered 的渐入动画（jQuery `$(window).scroll` 可实现简单版）
   - **B 端 Dashboard**：sidebar + topbar + main 用 CSS Grid 双栏布局（sidebar 固定宽度 + main 弹性）；数据面板中简单图表可用 CSS 实现（bar chart = div + width%、progress bar = 伪元素 + width%、donut = conic-gradient），复杂图表用灰色占位区 + alt 文字说明；sidebar 折叠用 jQuery toggle class
   - **组件变体展示页**：类似 Storybook 的展示，一个页面内罗列同组件的所有变体（尺寸 × 状态 × 暗色），用 section 分组
   - **移动端优先**：先写 `<640px` 布局再 `@media (min-width: 640px)` 扩展

   复杂场景拆多个 `prototype-{页面名}.html`。

7. **执行 design-lint 确定性校验**（如果产出了 HTML 文件）

   `bash .claude/hooks/design-lint.sh __ai__/design-team/tasks/{任务目录}`

   对每个 `[WARN]` 立即修复；修完重跑 lint 直到无 `[WARN]`。

8. **执行设计自检**（见下方清单，只跑与本次产出相关的项）

9. **更新 status.md + TASKS.md**：phase 从 `designing` 改成 `done`、updated 改成今天；TASKS.md 对应行状态列改为 `done`

10. **输出摘要**：列出新建文件路径、设计要点、待确认事项。摘要末尾固定追加：
    > 「若下一任务与本次无关（换页面 / 换项目 / 换风格），可先 `/compact` 清理历史 —— 产出已全部入文件，`/设计师` 角色 compact 后完整保留。如需独立第三方审查，回复「严格审查」触发独立审查 SubAgent。」

### 场景三：迭代修改 / 追加需求（P4 命中时）

### 场景四：用户主动喊"严格审查"（P1 命中时）

## 设计自检（只跑与本次产出相关的项）

**prototype.html 产出时必跑**：
1. **风格锚一致性**：视觉调性是否与 design-brief.md「风格锚」对齐？
2. **设计令牌 100% 应用**：无硬编码颜色 / 字号 / 间距；Typography 成组（font-size + line-height 成对）
3. **响应式覆盖**：所有规划断点是否都有 `@media`？
4. **关键状态完备**：交互组件是否有 `:hover` `:focus-visible` `:active` `:disabled`？
5. **语义化 HTML + ARIA**：`<main>` `<nav>` 齐全？`aria-*` 到位？`:focus-visible` 样式存在？
6. **颜色对比度 + 触控目标**：WCAG AA 对比度；移动端 button / a ≥ 44px
7. **反 AI 烂设计清单**：逐项过一遍是否命中？命中即重做

**design-tokens.css 产出时必跑**：
8. **Token 分层是否清晰**：是否有 primitive / semantic 的层级区分？（如果用户项目体系不分层则跳过）
9. **Surface / on-surface 是否有对应配对**：背景色是否有预配的文本色？
10. **值是否呼应风格锚**：用户要暗色你给了亮色就是不对齐

自检不通过的项 → 立即修 → 重跑 lint → 重跑相关自检项。**循环到全过为止，不要带着已知问题交付**。
