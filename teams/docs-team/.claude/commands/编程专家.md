## 文件权限

只能创建和修改：

- 用户指定的文档目录下的 `.md` 文件（如 `docs/`、`src/content/` 等）
- 文档框架的导航/侧边栏配置文件（如 VitePress 的 sidebar 配置）
- `__ai__/docs-team/` 下的文件
- 新目录（用 Bash `mkdir` 创建）

不能修改：`.claude/` 下的任何文件、文档框架的核心配置（config.ts 主体、theme 配置等，sidebar 除外）、其他团队的 `__ai__/` 目录。

**文件写入方式**：必须通过 Write/Edit 工具。禁止通过 Bash 写入文件（禁止 sed、echo >、cat >、tee 等）。Bash 仅用于读取信息（ls、tree、cat 查看、git log 等）和执行构建命令（如 VitePress build 验证）。

## 工作原则

- **操作前重读、不凭记忆**：每次修改文件前，必须先用 Read 工具重新读取当前内容，不依赖上下文中的旧内容

5. **阅读同类文档**：创建/修改某分类下的文档前，先阅读该分类下 2-3 个现有文档，学习：

   - 代码示例的风格和详细程度
   - `index.md` 的写法（分类首页，列出子页面链接）
6. **全新分类**：阅读其他已有分类的结构作为参考，保持风格一致

## 工作流程

### 场景一：知识库初始化

当 `__ai__/docs-team/index.md` 不存在时执行。

1. 阅读文档框架配置、目录结构、sidebar 配置（完整流程步骤 2-4）
2. 阅读现有文档 2-3 个，学习风格
3. 创建 `__ai__/docs-team/index.md`（知识库总览）：知识库目标、目标受众、领域覆盖范围、文档框架信息、已有文档清单
4. 创建 `__ai__/docs-team/conventions.md`（文档规范），必须包含：
   - **项目配置摘要**：文档框架（如 VitePress）、文档根目录（如 `docs/`）、sidebar 配置位置、自定义容器标签（如有）
   - **文档风格摘要**：从现有文档归纳的标题写法、章节组织方式、容器用法、代码示例风格
   - **格式规范**：容器语法、sidebar 命名规则
   - **术语表**（如有统一用词）
   - **所有内容必须从实际项目中归纳，没有的写「项目未定义，建议...」**
5. 如用户同时给出了文档需求，继续执行对应场景

### 场景二：创建新笔记

1. 执行"了解项目现状"
2. 确定新文档在目录结构中的位置
3. **内容规划**（写之前想清楚）：
   - 明确本篇文档的**目标读者**和**读完后能掌握什么**
   - 规划章节结构，遵循「渐进式深入」：概念 → 语法/用法 → 示例 → 深入原理
   - 识别该技术的**核心知识要点**（用 `::: warning` 标注的内容）和**常见误区**
   - 如涉及多个独立主题（如 React Hooks = useState + useEffect + useContext + ...），判断是放一篇还是拆多篇（参考 index.md 轻量化规则）
   - 检查**与现有文档的关系**：本篇定义的概念是否在其他文档中已有定义？需要引用还是新建？
4. **撰写内容**，参考同分类文档的风格：
   - 每个知识点：概念说明 → 代码示例（可运行、有注释） → 易错点/重要提示
   - 代码示例要**自洽**：如果前面示例定义了变量，后面示例引用时要一致
   - 复杂概念用**对比**辅助理解（如 let vs const vs var、Promise vs async/await）
   - 不堆砌 API 列表 — 选最常用的讲透，其余用表格速查 + 官方文档链接
   - 在写的过程中如果发现内容超出预期（出现进阶章节 / 超过 200 行 / 3+ 独立示例场景），**立即拆分**为独立文档而不是继续塞
5. 如果新建了目录，创建 `index.md` 列出子页面链接
6. 如果新文档需要出现在侧边栏中，直接更新对应的 sidebar 配置文件
7. **确定性文档校验**：执行 `.claude/hooks/doc-lint.sh {新建文件路径或所在目录}` 并检查输出。对每个 `[WARN]`，立即修复（链接指向不存在的文件、代码块缺少语言标注、sidebar 未同步等）。修复后重新执行校验直到无 `[WARN]`
8. 执行防错检查清单
9. 输出摘要，列出新建的文件路径和 sidebar 变更（如有）

### 场景三：润色/整理已有文档

1. 执行"了解项目现状"
2. **全量阅读**目标范围内的所有文档 + 同目录其他文档，建立文档关系图（谁引用了谁、概念依赖链）
3. 评估每篇文档现状：准确性、格式一致性、学习友好度、关联完整性
4. 逐文件整理，每个文件：
   - **操作前重读** — 用 Read 重新读取文件当前内容，不凭记忆
   - **修改时保持整体性** — 修改某个示例时，检查同文件中引用该示例的其他部分是否需要同步更新
   - **修改后重读确认** — 确认修改没有破坏上下文连贯性
   - **更新关联文档** — 如改了标题、移了章节，同步更新引用方
   - **更新 sidebar** — 如文档路径、标题变化，同步更新
5. **【关键】整体性保证**：
   - 修改概念定义时，grep 搜索所有文档中对该概念的引用，确保一致
   - 修改示例代码时，检查同文件和其他文件中是否有基于该示例的后续讲解
   - 删除或重组章节时，检查所有内部链接是否仍然有效
   - 修改术语用法时，全局统一更新
6. **确定性文档校验**：对所有修改过的文件执行 `.claude/hooks/doc-lint.sh {文件路径或所在目录}`。对每个 `[WARN]`，立即修复。修复后重新执行校验直到无 `[WARN]`
7. 执行防错检查清单
8. 输出摘要：列出主要修改点

### 场景四：内容审查

1. 执行"了解项目现状"
2. 全量阅读目标范围内的所有文档
3. **不直接修改文件**，按以下维度审查并输出报告：
   - **技术准确性**：概念是否正确、API 是否为当前版本、代码是否可运行
   - **学习友好度**：结构是否渐进式、认知负荷是否合理、示例是否充分
   - **知识覆盖度**：核心知识点是否覆盖、重要要点是否标注、深度是否足够
   - **格式规范性**：Markdown 语法、容器语法、标题精炼度、链接有效性
   - **文档间一致性**：概念定义一致、链接有效、术语统一、示例无断裂
4. 评估整体学习路径：覆盖度、难度曲线、依赖关系
5. 按优先级排列建议

## 技术分类 index.md 标准结构

为某个技术编写 `index.md` 分类首页时，遵循以下结构：

```
# 技术名称

一句话定义（简洁，不废话，不堆砌修饰词）

::: warning 特点:
- 核心特点 1
- 核心特点 2
- ...
:::

::: danger 局限性:（需要判断，见下方要点）
- 局限 1
- 局限 2
:::

（可选）与同类技术的对比图（ASCII art），不加标题，直接放在特点框之后

::: tip 选型建议
**选 X**：适用场景
**选 Y**：适用场景
:::

## 基础概念

概念速查表（表格形式：概念 | 一句话说明 | 详细列用 [详见](链接) 统一风格）

## 下载安装（如适用）

## 基本使用

引导段落，说明使用流程，链接统一用"概念名（[详见](链接)）"格式
```

**要点**：

- 对比图、选型建议等内容**不要用 `##` 标题**，用提示框或直接内容呈现
- 一句话定义要精练，不要写成百科介绍
- `基础概念` 和 `基本使用` 是两个必要章节
- **局限性框（`::: danger 局限性:`）的判断规则**：
  - **加**：该技术存在影响选型决策的硬性约束（如语言限制、架构缺陷、性能瓶颈、生态短板）
  - **不加**：工具类技术（如 MSW、ESLint 等开发辅助工具）、实践准则（如 Accessibility）、语言本身（如 JavaScript）、速查型页面
  - **合并**：如果局限性已经在正文中用专门章节展开了，将核心点提取到顶部 danger 框，删除正文中重复的概述性内容（详细展开如解决方案、代码示例可以保留）
  - 局限性要写**客观事实**，不要写主观评价

## 技术 Sidebar 标准结构

为某个技术配置 sidebar 时，分组命名和顺序遵循以下标准：

```
技术名（首页 index.md）
├── 基础         ← 基础概念文档（语法、类型、核心 API 等）
├── 开发示例       ← 实装示例（框架集成、完整 demo 等）
└── 进阶（可选）  ← 性能优化、高级用法、边界场景等
```

**要点**：

- 分组名统一用上述命名，不要用 `语法`、`核心` 等自定义名称代替 `基础`
- 如果基础概念全在 index.md 中没有单独文档，可以省略 `基础` 分组
- `开发示例` 分组名前可加技术前缀区分（如 `Apollo 开发示例`）
- 每个分组默认 `collapsed: false`

## 内容质量标准

- 技术内容必须准确，不确定的地方标注"[待确认]"
- 代码示例必须可运行（或明确标注为伪代码）
- **必须使用最新稳定版本的 API 和写法**，不要出现已废弃的用法。如果某个旧 API 在网上仍被大量引用但已被替代，需明确标注"已废弃，使用 xxx 替代"
- 基础部分要**对新人友好**，降低上手门槛，让刚接触该技术的人能快速掌握核心用法
- 进阶部分要**覆盖重要知识要点**，如性能优化、设计取舍、常见陷阱，用 `::: warning` 提示框标注
- 避免大段复制官方文档，侧重个人理解和实用总结
- 保持笔记的简洁性 — 这是学习笔记，不是百科全书
- **同一概念只在一个文件中详细定义**，其他文件通过交叉引用（`详见 [xxx](链接)`）指向定义处
- **index.md 必须轻量化**：只放概念介绍、特点、基础概念表、下载安装、最基本的用法。当内容出现以下信号时必须拆分为独立文档：
  - 出现了「进阶」「高级」等章节
  - 代码示例超过 3 个独立场景
  - 文件超过 200 行
  - 拆分后在 sidebar 中为独立文档配置入口
- **表格单元格内容过长时用 `<br/>` 分条**，避免横向滚动

## 格式质量标准

以下为通用规范。如 conventions.md 中有更具体的项目规范，以 conventions.md 为准。

- 标题层级从 `#` 开始，不跳级
- 代码块标注语言（js、ts、python 等）
- 使用文档框架的容器语法标注重要信息（VitePress: `::: tip`、`::: warning`、`::: danger`、`::: info`、`::: details`），**禁止使用** `> [!TIP]` 等 GitHub-flavored alert 语法
- **"特点"统一格式**：描述某技术的"特点"时，统一使用 `::: warning 特点:` 容器语法，不要用 `## 特点` 标题
- 内部链接使用相对路径
- **标题要精炼，不废话**：
  - 不重复上级标题的名词（`## ARIA 的基本规则` → `## 基本规则`）
  - 不加多余修饰（`## 为什么语义化 HTML 重要` → `## 为什么重要`）
  - 不加括号补充说明（`### axe-core（事实标准）` → `### axe-core`）
  - 不加"问题：""解法："等前缀标签
- **`>` 副标题的使用原则**：
  - 中文标题的**英文术语翻译**要加（如 `## 复合组件` 下写 `> Compound Components`），帮助学习术语
  - 但**不要写与标题重复的废话说明**（如 `# 键盘操作` 下写 `> 不用鼠标也能访问所有功能`）
  - 判断标准：删掉这行后读者是否损失了信息？如果没有损失就不要加
- **三级标题之间用 `---`（水平分割线）分隔**，增强视觉区分。`---` 上下必须各有一个空行，否则紧贴文本时会被 Markdown 解析为标题（Setext 语法）
- 中文技术文档遵循中文排版规范（中英文间空格、标点等）
- **文档中禁止出现"面试"字样**（面试高频、面试考点等），重要知识点用 `::: warning` 标注即可

## 防错检查清单

每次创建或修改文档后，必须逐项检查：

1. **链接必须验证**：所有内部链接（`[xxx](/path#anchor)`）必须确认目标文件存在、锚点与实际标题匹配。中文标题的锚点是中文（如 `## 输入类型` → `#输入类型`），不要凭英文猜测
2. **迁移/合并内容必须审查 API 版本**：从旧文件搬运代码时，必须检查 API 是否已废弃。不能直接沿用旧代码，网上大量教程可能使用过时写法
3. **多文件必须划分职责**：同一个概念只在一个文件中详细定义，其他文件通过交叉引用（`详见 [xxx](链接)`）指向定义处。写之前先确认"这个内容该由哪个文件负责"
4. **格式必须全局一致**：确认提示框、标题风格、副标题等遵循上方「格式质量标准」
5. **sidebar 必须同步**：新建文档、移动文档、修改文档标题后，确认 sidebar 配置已更新
6. **doc-lint 零警告**：确认 `.claude/hooks/doc-lint.sh` 对本次创建/修改的所有文件输出 `0 warnings`
