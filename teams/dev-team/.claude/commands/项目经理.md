# 项目经理（调度器）

调度 项目经理 + 验证官 完成设计。调度器只做编排，不写业务代码/设计文档，但可读写 status.md（phase/rework_reason）和 TASKS.md（状态列、文档同步列）。

---

## 步骤

### 1. 确定任务与场景

#### 1.1 扫描候选关联任务集

- 如 `__ai__/dev-team/tasks/TASKS.md` 不存在 → 候选集为空，跳到 1.2
- 否则读取 TASKS.md，提取「状态」列 ∈ 未完成状态集的行：
  `{designing, verified, developing, dev-done, reviewing, review-passed, testing, rework}`
  （即除 `done` 外的全部状态）
- 用 `git branch --show-current` 取当前分支，在候选集中优先保留「分支列 = 当前分支」的行

#### 1.2 按优先级判定场景（第一条命中即停）

触发词：
- 追加词：`再加` `还要` `补充` `顺便` `另外`
- 修正词：`改成` `改为` `不对` `应该是` `有问题` `重做` `换成`
- 指代词：`这个` `刚才那个` `那个任务`
- 新开词：`新开` `另起` `单独做` `新任务`
- 接手词：`接手` `分析这个分支`
- 同步词：`同步文档` `更新文档` `更新 index` `更新项目文档`

1. **P1** 同步词 → 场景 5（不建目录）
2. **P2** 接手词 → 场景 4（新目录含 `_接手` 后缀）
3. **P3** 新开词 → 场景 2 新任务
4. **P4** 候选集非空 + 含追加/修正/指代词 → 场景 3
   - 候选 1 项：定位该目录
   - 候选多项：列出 `| 序号 | 目录 | 分支 | phase |` 请用户重新调用时明确归属，停止执行
5. **P5** 无需求描述 + 候选集非空：
   - 有 `phase=designing` 任务：定位该目录（场景 2 续作）
   - 否则：列候选让用户选（1.4）
6. **P6** 有需求描述且以上未命中 → 场景 2 新任务
7. **P7** 无需求描述且候选空 → 提示描述需求，停止

#### 1.3 路径处理

- **定位已有**（P4/P5）：`TASK_DIR=候选目录`
- **新建目录**（P2/P3/P6）：从用户消息提功能名（无法提取用 `未命名`），路径 `__ai__/dev-team/tasks/YYYYMMDD_{功能名}_{后缀}/`；若已存在加 `_2/_3/...` 后缀；`mkdir -p`
- **不建目录**（P1）：跳过

#### 1.4 歧义兜底

多候选、触发词同时命中多优先级时，停止执行，用 `[场景待确认]` 列出可能场景请用户明确。

### 2. 更新 phase

- **场景 5**：跳过
- **场景 2/4（新建）**：按 templates/status.md 写 status.md `phase: designing, updated: 今天`；TASKS.md 新增行 `| 目录/ | 正式任务 | 分支 | designing | [ ] |`（不存在按 templates/tasks.md 创建）
- **场景 2 续作（P5）**：仅更新 status.md `updated`
- **场景 3（变更）**：
  - 读出原 phase（需要保留到步骤 4 供回滚用）
  - 把 status.md 改为 `phase: designing, updated: 今天`；若原 phase=`rework` 同时清 `rework_reason: ""`
  - TASKS.md 状态列改回 `designing`，文档同步列改 `[ ]`
  - 原 phase + 用户消息片段通过 `[场景3上下文]` 传给 agent

### 3. spawn 项目经理

spawn name=`项目经理`，prompt 必含以下字段（不适用的字段整行省略）：

```
[调度器判定] 场景 = {2 / 3 / 4 / 5}
[任务目录] {TASK_DIR}    # 场景 5 省略
[用户原始消息] {$ARGUMENTS}
[已有产出] {目录下已有文件列表}    # 仅场景 2 续作/场景 3 输出
[场景3上下文] 原 phase={X}, 触发片段={Y}    # 仅场景 3 输出
[执行指令] 按场景 {N} 流程执行，禁止自行判定场景；禁止重复改 TASKS.md 的状态列（场景 5 可正常更新文档同步列）
```

条件追加：
- 场景 2 且 index.md 不存在：「先执行场景 1 初始化，再处理任务」
- 场景 2 续作：「在现有 design.md 上迭代，不整份覆盖」
- 场景 3：「禁止新建目录；若需求与任务功能域无关，摘要首行输出 `[场景疑似误判]` 说明理由且不修改任何文件」
- 返回 ≤200 字摘要

### 4. 事后验证（项目经理）

`git diff --name-only` + `git status --short` 检查只修改了 `__ai__/dev-team/` 下文件，越权则警告并建议 revert。

**场景 3 疑似误判回滚**：若 agent 摘要首行含 `[场景疑似误判]`，把 status.md 的 phase 改回原值、TASKS.md 状态列与文档同步列恢复，向用户说明 `[场景 3 回滚] 已恢复原 phase={X}，如需新开请回复「新开」`，停止后续。

### 5. 产物校验

场景 5 跳过本步及 5.5/5.6/6-8，直接到步骤 9。

检查任务目录下 `design.md` 和 `dev-tasks.md` 是否存在。

- **任一不存在** → 向用户报告缺失文件，跳过验证官，结束
- **均存在** → 用 grep 检查 design.md 关键章节完整性：
  - `现有代码分析` — 缺失则警告"design.md 缺少现有代码分析"
  - `对现有功能的影响分析` — 如涉及现有功能修改（非纯新增项目），缺失则警告"design.md 缺少影响分析"
  - 警告不阻塞流程，但展示给用户知晓

### 5.5 影响分析辅助验证

调度器主上下文执行：从 design.md「需要修改的部分」提取文件路径，对每个执行 `grep -rl "{文件名}" --exclude-dir=__ai__ --exclude-dir=.claude --exclude-dir=node_modules --exclude-dir=.git .`，汇总后附给步骤 6 验证官 prompt。

> 局限：grep 只能发现直接引用，间接/运行时依赖不覆盖。

### 5.6 确定性文档校验

执行 `.claude/hooks/doc-lint.sh {任务目录}`，捕获 `[WARN]`/`[INFO]` 输出；有 WARN 则附给步骤 6 验证官 prompt（标注「确定性校验结果」），同时展示给用户。

### 6-8. 验证循环（最多 3 轮，verify_round=1 起）

#### 6. spawn 验证官

spawn name=`验证官`，prompt 包含：任务目录、轮次、5.5+5.6 结果（首轮）、非首轮附上轮 fail 原因、要求 ≤200 字摘要。

#### 7. 事后验证（验证官）

`git diff --name-only` 检查验证官只改了任务目录下 `design.md` 和 `dev-tasks.md`，越权警告。

#### 8. 读取 verdict 并决定是否循环

用 grep 读取 `design.md` 中 `verdict:` 行：

- **pass / pass-with-notes** → 退出循环，status.md `phase: verified`
- **fail 且 round < 3** → 简报验证结果；spawn 项目经理 rework，prompt 沿用原场景标注 `[调度器判定] 场景={保持原值}`、`[任务目录]`、`[执行指令] rework 第 N 轮，基于验证官反馈改进，禁止新建目录`；完成后重做步骤 4+5+5.6，round+=1 回步骤 6
- **fail 且 round >= 3** → 停循环，phase 保持 `designing`，告知用户 3 轮未通过需人工介入

### 9. 收尾

- 场景 5：agent 已更新 TASKS.md 文档同步列；展示摘要，结束
- 场景 2/3/4：更新 TASKS.md 状态列（`designing`→`verified` 或保持 `designing`）；verified → 提示 `/程序员 {任务目录}`；否则提示查看验证报告后重调本命令
